# 三方支付数据统计系统 — 产品原型规划

## Context

财务/风控部门需要识别和监控企业认证客户的订单收款中是否存在第三方支付情况。当前缺乏一个可视化的统计工具来逐笔展示订单的收款明细，并清晰标识已收款/未收款、三方支付、个人报销等场景。本原型旨在为"三方支付数据统计系统"提供一个完整的交互式产品演示，以订单收款明细为核心，每一笔收款独立展示，未收款订单展示应收金额。

## Recommended Approach

### 1. 整体架构

新建单文件 HTML 原型：`d:/fsy/99.其他/TYLM/三方支付数据统计原型.html`

- 纯前端实现，依赖 Tailwind CSS v3 + Font Awesome 6（CDN）
- 采用左右分栏布局：左侧 78%-80% 产品界面，右侧 20%-22% 原型说明
- SVG 贝塞尔曲线连线（左侧 3 个关键元素 ↔ 右侧 3 个说明卡片）
- 悬停联动高亮、响应式适配（PC 左右分栏 / 移动端上下布局）

风格参考：
- `ecommerce-admin-prototype.html`：SVG 连线系统、Tailwind 混合架构、响应式处理
- `付款管理原型.html`：华秋系统红色头部（#c41e24）、侧边菜单、筛选布局、表格样式、弹窗交互

### 2. 页面结构（从左到右、从上到下）

#### 2.1 顶部导航栏
- 华秋系统红色背景（#c41e24），系统名称"三方支付数据统计系统"，当前激活"三方支付统计"

#### 2.2 侧边菜单
- 分组：数据统计（数据仪表盘、三方支付统计[active]）、客户管理、系统设置

#### 2.3 筛选区域（proto-element id=1）
- 白色圆角卡片内，两行筛选条件
- 条件：时间范围（默认 2026-01-01 ~ 今天）、业务线、线上/线下、境内/境外、是否收款、是否三方支付、是否个人报销、客户属性、客户名称搜索
- 操作：查询、重置、导出按钮

#### 2.4 订单收款明细列表（proto-element id=2）
- 核心数据区，以订单收款记录为维度，每笔收款独立一行，未收款订单也展示一行
- 约 18-20 列，支持水平滚动
- 列定义：
  - 年份
  - 订单号
  - 华秋ID
  - 客户名称（按优先级取值后的名称）
  - 线上/线下
  - 境内/境外
  - 企业认证
  - 客户属性（公司/个人）
  - 业务线
  - 订单金额
  - 是否收款（Tag：绿色-是，灰色-否）
  - 应收金额（未收款时显示订单应收金额，已收款时可为空或显示"—"）
  - 订单支付方式（取主订单的支付方式）
  - 付款名称（已收款时按规则取值，未收款时为空）
  - 付款方式（已收款时的实际收款方式，未收款时为空）
  - 支付流水号（已收款时，未收款时为空）
  - 第三方支付流水号（已收款时，未收款时为空）
  - 实际付款金额（已收款时，未收款时为空）
  - 开票名称
  - 个人报销（Tag：橙色-是，灰色-否/空；未收款时为空或"未收款"）
  - 三方支付（Tag：红色-是，灰色-否/空；未收款时为空或"未收款"）
- Tag 颜色规范：三方支付-红色、个人报销-橙色、企业认证-绿色、线上-蓝色、线下-灰色、已收款-绿色、未收款-灰色
- 分页组件

#### 2.5 规则说明入口（proto-element id=3）
- 表格上方的小型信息卡片，展示核心判定规则缩略（客户名称优先级、三方支付判定、付款名称取值规则），点击/悬停联动右侧说明

#### 2.6 右侧说明面板
- 琥珀色系背景（#fffbeb）+ 左侧琥珀色边框
- 3 个说明卡片对应 3 条 SVG 连线：
  1. 多维度筛选查询说明（含新增的是否收款筛选）
  2. 订单收款明细列表说明（以订单+收款记录为维度，每笔收款独立一行；未收款订单展示应收金额）
  3. 核心判定规则说明（三方支付定义、线上/线下定义、客户名称优先级、付款名称取值规则、个人报销判定、名称是否为公司）

### 4. 关键字段取值规则（已与用户核对确认）

#### 4.1 基础订单字段

| 字段 | 取值规则 |
|------|----------|
| 年份 | 按应收生成的年份 |
| 订单号 | 按系统订单号 |
| 华秋ID | 按订单的华秋ID |
| 金额（订单金额） | 按订单的金额 |
| 业务线 | 按订单的业务线（PCB / PCBA 等） |

#### 4.2 客户相关字段

| 字段 | 取值规则 |
|------|----------|
| 客户名称 | **优先级**：合同名称 > 发票开票抬头 > 企业认证名称 > 月结名称 > 在线账号名称，优先取到公司名称 |
| 线上/线下 | 前台下过单 → **线上**；pcb/pcba提供的客户名单 → **线下**；一个客户同时存在 → **归为线下** |
| 境内/境外 | 真实收货地：中国大陆 → **境内**；港澳台及国外 → **境外** |
| 企业认证 | 按CRM客户认证数据 |
| 客户属性（公司/个人） | 企业认证客户 或 信用客户（信用账期+正常/冻结状态）→ **公司**；否则 → **个人** |

#### 4.3 收款相关字段

| 字段 | 取值规则 |
|------|----------|
| 是否收款 | **实际付款金额 > 0 → 是；付款金额 = 0 或无收款记录 → 否** |
| 应收金额 | 未收款时 = **订单应收金额**；已收款时显示 **—** 或留空 |
| 订单支付方式 | **取主订单的支付方式**（订单维度，同一订单的多行收款记录值相同） |
| 付款名称 | 见下方「付款名称取值规则」 |
| 付款方式 | 实际收款方式（公司转账、汇付天下、余额支付、信用支付、微信支付、支付宝、银联等），有多笔则体现多笔 |
| 支付流水号 | 对应付款方式的流水号，有多笔则体现多笔 |
| 第三方支付流水号 | 对应付款方式的第三方流水号，有多笔则体现多笔 |
| 付款金额 | 实际付款金额，有多笔则体现多笔 |
| 开票名称 | 系统有效发票的开票抬头，多个用 **,** 隔开 |

#### 4.4 判定字段

| 字段 | 取值规则 |
|------|----------|
| 个人报销 | 同时满足：① 支付方式为 **微信/支付宝/汇付天下/银联**；② **付款人为空**（取不到付款人）→ **是**；否则 **否** |
| 三方支付 | 客户名称 与 付款名称 **不一致** → **是**；**一致** 或 **付款人为空** → **否**。微信/支付宝/汇付天下/银联且付款人为空的情况，优先判定为「个人报销」，不归为三方支付 |

#### 4.5 付款名称取值规则

| 支付方式 | 付款名称取值逻辑 |
|----------|-----------------|
| **公司转账** | 取真实打款人名称。找不到的情况（手动核销无备注、历史久远、信用支付未还款或退款抵扣）→ **空** |
| **汇付天下** | 取财务台账的付款人 |
| **余额支付** | ① 所有充值记录均为公司转账/汇付天下 → 取订单付款时间之前的**最后一笔充值单**的付款人<br>② 充值记录混合（含微信/支付宝等）→ 取**两个值**：a) 最后一笔公司转账/汇付天下充值单的付款人；b) 最后一笔充值单的付款人。表格中**分两行展示** |
| **信用支付** | ① 信用还款方式为**余额支付** → 按余额支付规则处理<br>② 其他方式还款 → 给出还款支付方式对应的公司名（公司转账）或汇付天下流水号（汇付天下）<br>③ **未还款** → 显示 **"未还款无付款人"** |

#### 4.6 名称是否为公司（辅助判断）

名称包含以下关键词之一则为**公司**，否则为**个人**：
公司、有限、责任、集团、股份、厂、店、行、社、企业、工作室、中心、部、局、处、委员会、协会、商会、联社、总会、分会

关键设计点：
- 同一个订单有多笔收款的，该订单出现多行（如订单E有两笔余额支付、订单N有两笔公司转账），每笔展示各自的付款名称和付款金额
- **是否收款判定**：实际付款金额 > 0 → 是；付款金额 = 0 或无收款记录 → 否
- 未收款订单出现一行，是否收款=否，应收金额=订单金额，收款相关列为空
- 已收款但找不到付款人的（如手动核销无备注），付款名称为空，个人报销=是
- 未还款的信用支付：付款金额=0 → 是否收款=否，应收金额=订单金额，付款名称="未还款无付款人"

### 6. JavaScript 交互逻辑

- **筛选联动**：筛选条件变化实时触发过滤并渲染表格
- **表格渲染**：分页逻辑（每页 10-15 条），动态生成 tbody；同一订单的多笔收款在表格中连续展示，可用背景色或左侧边框 subtle 区分
- **SVG 连线**：DOM 渲染完成后绘制 3 条贝塞尔曲线（筛选区、明细列表、规则入口 ↔ 右侧对应说明）；resize / scroll 时重绘；悬停联动高亮
- **响应式**：移动端切换为上下布局，连线隐藏，提供浮动切换按钮
